# LocalSweep — 产品需求文档

## 产品定位

面向本地 AI / 开发环境的 macOS 桌面工具，核心目标不是“系统优化”，而是：
1. 把正在监听端口、占用资源的服务和进程尽可能看清楚
2. 用更好的可视化信息帮助判断哪些可以结束
3. 将结构化列表交给 AI 做建议排序和风险解释
4. 由用户勾选后安全结束进程
5. 额外提供独立的缓存清理板块，解决磁盘空间被缓存长期占用的问题

核心痛点：
1. 本地模型服务、开发服务、测试服务经常残留监听端口，只知道端口号，不知道是谁
2. 机器发热、变卡时，很难快速定位真正该处理的进程
3. 现有工具的可视化信息不完整，没有图标、来源、项目归属、风险提示
4. 缓存长期堆积，占空间，但普通清理工具粒度过粗、风险不清楚

---

## 产品范围

### V1 核心范围

V1 仅包含两个板块：
1. 端口 / 进程清理
2. 缓存清理

AI 分析属于“端口 / 进程清理”板块内的能力，不单独作为一级页面。

### V1 非目标

以下内容不在 V1 范围：
- 自动执行 AI 建议
- 无确认直接结束进程
- 任意路径文件清理
- 清理日志、`/tmp`、`/var/tmp`、模拟器数据、废纸篓
- 做成通用系统监控面板
- 长时后台守护、自动巡检、定时任务

## 开源缝合策略

本项目采用“高星项目经验借鉴 + 选择性 fork + 本地缝合实现”的落地方式，不从零设计所有能力，也不直接拼装多个整仓成品。

### 1. 主基座策略

优先将以下项目作为能力来源：
- `neohtop`
  - 角色：桌面应用结构、暗色主题系统监控 UI 组织、信息密度设计参考
  - 使用方式：优先作为产品壳层与交互组织参考，不要求沿用其前端框架
- `bottom`
  - 角色：进程列表排序、搜索、资源展示、结束信号交互经验
  - 使用方式：重点借鉴交互模型、字段组织和 kill 流程，不直接作为 GUI 基座
- `rustnet`
  - 角色：网络连接与进程关联、系统网络信息抓取思路
  - 使用方式：重点借鉴端口到进程关联的数据采集与模型设计
- `osx-cleaner`
  - 角色：macOS 缓存扫描与安全边界
  - 使用方式：只参考缓存清理边界、扫描/清理流程与风险控制，不作为主基座

### 2. 具体落地原则

- 不粗暴整仓合并多个项目，避免技术栈和架构风格冲突
- 优先复用“数据模型、交互模式、能力边界”，其次才考虑复制实现代码
- 若上游项目与当前技术栈高度一致，允许做单项目轻量 fork，再在本仓内继续定制
- 若上游项目是 TUI 或 CLI 工具，只借鉴逻辑和实现思路，不强行改造成 UI 主体
- 缓存清理保持自研实现，避免把低相关度系统工具整体引入主仓

### 3. 当前推荐路线

- 主基座：`neohtop`
- 进程交互设计参考：`bottom`
- 端口 / 进程关联参考：`rustnet`
- 缓存清理安全边界参考：`osx-cleaner`

## 核心板块

### 一、端口 / 进程清理

这是产品主板块，目标是将“监听端口”和“运行进程”整合成一份可操作的候选列表。

#### 1. 数据采集

系统需采集并整合以下信息：
- 监听中的端口列表，仅关注监听态，不展示瞬时连接态
- 端口关联的进程信息：进程名、PID、可执行文件路径、命令行摘要
- 资源占用：CPU、内存、启动时间、进程状态
- 端口信息：端口号、协议、同一进程监听的全部端口
- 补充信息：应用图标、应用来源、推测项目目录、风险标签

#### 2. 信息补全

每一项候选对象应尽量补齐以下字段：
- 图标
- 进程名
- PID
- 监听端口列表
- 可执行路径
- 命令行摘要
- 推测项目目录
- CPU 占用
- 内存占用
- 启动时间
- 风险标签

补充规则：
- 图标优先从 `.app` bundle 或可执行文件关联信息中提取，失败时降级为默认图标
- 项目目录向上查找 `package.json`、`pnpm-lock.yaml`、`Cargo.toml`、`go.mod`、`.git`
- 项目目录推测优先级：`package.json`/`pnpm-lock.yaml` > `Cargo.toml` > `go.mod` > `.git`
- 最多向上回溯 6 层目录，超出后停止，避免整盘扫描
- 命令行过长时只展示摘要，完整内容放入详情区

#### 3. 展示视图

同一份底层数据提供两个视角：

端口视图：
- 适合回答“某个端口是谁占用的”
- 默认按端口号排序
- 支持按端口号、进程名、路径、项目目录过滤

进程视图：
- 适合回答“谁最占 CPU / 内存”
- 支持按 CPU、内存排序
- 支持按进程名、路径、项目目录过滤

两种视图共享同一份候选对象，避免出现重复维护的两套状态。

#### 4. AI 分析

用户可对以下范围发起分析：
- 当前全部候选项
- 当前筛选结果
- 当前勾选结果

AI 输入不是自由文本，而是结构化候选列表。AI 只负责：
- 总结当前异常情况
- 标记建议优先处理的候选项
- 给出原因说明
- 给出风险等级

AI 不负责直接执行任何动作。

#### 5. 勾选与结束进程

执行流程：
1. 用户查看列表或先执行 AI 分析
2. 系统根据 AI 返回结果高亮推荐项
3. 用户勾选要处理的项
4. 用户二次确认
5. 系统先执行温和终止
6. 若超时未退出，再提示是否强制结束
7. 执行后刷新列表并输出结果

支持：
- 单项结束
- 多项勾选后批量结束
- 展示每项执行结果

约束：
- 对系统关键进程、当前应用进程、父级守护进程默认禁用结束操作
- 若结束失败，需返回明确原因，如权限不足、进程已退出、信号发送失败
- AI 推荐只影响默认勾选建议，不得跳过用户确认直接执行

### 二、缓存清理

缓存清理是独立板块，不与端口 / 进程清理共用流程。

#### 1. V1 清理范围

V1 仅处理以下缓存目录：
- 应用缓存：`~/Library/Caches/`
- Xcode 缓存：`~/Library/Developer/Xcode/DerivedData/`
- Cargo 缓存：`~/.cargo/registry/cache/`

#### 2. 交互流程

执行流程：
1. 扫描缓存目录
2. 展示各类别路径、文件数量、总大小
3. 用户勾选要清理的类别
4. 用户二次确认
5. 执行清理
6. 输出结果报告

#### 3. 约束

- 首版采用“先扫描、后清理”两阶段流程，禁止未扫描直接删除
- 每个类别需先展示命中路径、文件数量、总大小，再允许勾选清理
- 默认不跟随符号链接，不清理受 SIP 保护路径，不越过用户主目录边界
- 清理完成后输出结果报告：成功删除数量、释放空间、失败路径及失败原因
- 不支持自定义任意路径清理，避免功能越界

## 统一数据模型

### 候选对象

端口视图和进程视图共享同一份候选对象，建议字段如下：

```json
{
  "id": "proc_12345",
  "processName": "node",
  "pid": 12345,
  "icon": null,
  "ports": [
    {
      "port": 3000,
      "protocol": "tcp"
    }
  ],
  "executablePath": "/usr/local/bin/node",
  "commandSummary": "node server.js",
  "projectPath": "/Users/name/work/demo-app",
  "cpuPercent": 82.4,
  "memoryMb": 612,
  "startedAt": "2026-03-19T11:20:00+08:00",
  "status": "running",
  "riskTags": ["dev-service", "high-cpu"]
}
```

### AI 返回结构

AI 返回必须是结构化数据，建议如下：

```json
{
  "summary": "当前共有 4 个候选项值得优先处理，其中 2 个属于高占用开发服务残留。",
  "suggestions": [
    {
      "candidateId": "proc_12345",
      "recommendation": "kill",
      "reason": "该进程持续占用 3000 端口且 CPU 高，占用来源为本地开发目录。",
      "riskLevel": "medium"
    }
  ]
}
```

## 页面结构

顶部导航采用 3 个一级页面：
1. 端口 / 进程
2. 缓存清理
3. 设置

其中：
- `端口 / 进程` 页面内提供 `端口视图` 和 `进程视图` 两个切换视角
- AI 分析入口位于 `端口 / 进程` 页面，不单独占一个页面
- 设置入口也可放在右上角，但底层仍归属于一级页面

整体风格：
- OpenAI 风格的克制工作台
- 圆角卡片
- 信息密度高，但重点信息需突出
- 图标、进程名、端口、资源占用需优先进入首屏视野

## 安全与交互规范

- 所有高风险操作必须二次确认，包括：结束进程、强制结束进程、执行缓存清理
- 确认信息至少包含：操作对象、影响范围、潜在风险
- 对高风险失败结果必须保留可追溯记录，至少包含时间、对象、结果、失败原因
- 清理与结束进程的执行入口必须区分“建议操作”和“真实执行”，避免用户误触
- 若需要更高权限，需明确提示系统授权原因；授权失败时保持只读，不做降级删除或强制重试
